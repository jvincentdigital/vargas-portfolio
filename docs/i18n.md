# i18n

The site ships in Spanish (default) and English via [`next-intl`](https://next-intl.dev/).

## How routing works

- Locales are declared in `src/i18n/routing.ts`:
  ```ts
  locales: ['es', 'en']
  defaultLocale: 'es'
  ```
- `src/middleware.ts` runs `next-intl/middleware` with matcher `['/', '/(es|en)/:path*']`. A request to `/` gets rewritten to the negotiated/default locale prefix.
- `src/app/[locale]/layout.tsx` reads `params.locale`, validates it against `routing.locales` (404 otherwise), then wraps children in `NextIntlClientProvider`.
- `src/i18n/request.ts` is the server-side config hook — it imports `messages/<locale>.json` per request and returns it to the provider.

## How components consume copy

Section components are `'use client'` (required by Framer Motion). They pull strings via:

```tsx
const t = useTranslations('about');
// t('title'), t('bio'), t('certifications_title')
```

For arrays (e.g. bullet lists), use `t.raw('certifications')` and map over the result.

## Adding / editing copy

1. Edit **both** `messages/es.json` and `messages/en.json`. Keep keys identical across files — missing keys throw at runtime.
2. Namespace keys by section (`nav`, `hero`, `about`, `skills`, `experience`, `portfolio`, `contact`, `footer`) so `useTranslations('<section>')` in the matching component picks them up cleanly.
3. Never hardcode user-facing strings in components. The only hardcoded strings that belong in TSX are contact info (email, phone, LinkedIn URL) — those don't translate.

## Adding a new locale

1. Add the locale code to `routing.ts` → `locales`.
2. Create `messages/<locale>.json` by copying `es.json` and translating values.
3. Update the middleware matcher in `src/middleware.ts` to include the new code, e.g. `/(es|en|fr)/:path*`.
4. Optional: set `defaultLocale` if the new locale should take over the root `/` route.

## Gotchas

- The `[locale]` layout receives `params` as a **Promise** in Next.js 14+ — always `await` it.
- If you see `NEXT_NOT_FOUND` on `/`, the middleware matcher is probably wrong.
- Google Fonts are loaded via `@import` in `globals.css`, not via `next/font`. This means no locale-specific font subsetting — both locales share the same font stack.
